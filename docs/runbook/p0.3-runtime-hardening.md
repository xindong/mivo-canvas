## 1. pm2 看护(ecosystem.config.cjs)

> 产物:`ecosystem.config.cjs`(仓库根)。看护项:max_memory_restart(内存看护)+ max_restarts/min_uptime/exp_backoff(重启风暴保护)+ 日志轮转。

### 1.1 三项看护(均已在 ecosystem.config.cjs 固化)

| 看护项 | 配置 | 触发即行为 | 依据 |
|---|---|---|---|
| 内存看护 | `max_memory_restart: '1G'` | BFF 超过 1G 即 pm2 自动 restart | 防 Leafer/Kysely/pg 累积内存泄漏拖垮进程;BFF 与 PG 容器(2G 限)独立 |
| 重启风暴保护 | `min_uptime: '15s'` + `max_restarts: 10` + `restart_delay: 3000` | 启动后活不到 15s 视为异常;窗内重启超 10 次 → pm2 停止重启(标 errored) | 防 crash-loop 反复重启打满 CPU/日志;依赖(PG/网关)挂时不无限打 |
| 指数退避 | `exp_backoff_restart_delay: 100` | crash 重启间隔 `100×2^n` 翻倍(pm2 自动封顶 15000ms) | 给依赖恢复时间,不是固定 3s 死磕。**必须用数值字段**(对象形式会被 pm2 忽略,见 F5 返修) |

### 1.2 日志轮转(pm2-logrotate 模块)

ecosystem 固化了日志路径(`/AIGC_Group/mivo-canvas-data/pm2/mivo-canvas.{out,err}.log`)+ `merge_logs` + 时间戳前缀。文件轮转由 **pm2-logrotate 模块**负责(ecosystem 本身不设轮转大小),服务器**一次性安装**:

```bash
# 以 yanjian 身份(pm2 主)安装 pm2-logrotate 模块(全局一次)
sudo -u yanjian pm2 install pm2-logrotate
# 轮转策略(示例;按服务器磁盘调):
sudo -u yanjian pm2 set pm2-logrotate:max_size 50M       # 单文件超 50M 轮转
sudo -u yanjian pm2 set pm2-logrotate:retain 14           # 保留 14 份
sudo -u yanjian pm2 set pm2-logrotate:compress true       # 轮转后 gzip
sudo -u yanjian pm2 set pm2-logrotate:dateFormat YYYY-MM-DD_HH-mm-ss
```
预期:`pm2 list` 见 `pm2-logrotate`;日志文件超 50M 自动切 `mivo-canvas.out.log__2026-07-12__03-00-00.gz`。

### 1.3 首次启动 / 更新(服务器,以 yanjian)— F6 返修:可执行部署链

**痛点(双审 F6)**:CLAUDE.md 权威四步只写 `pm2 restart mivo-canvas`(按名 restart),但:
1. **密码不映射**:`ops/postgres/.env` 的 `POSTGRES_PASSWORD` ≠ BFF 读的 `MIVO_PG_PASSWORD`。deploy.sh 必须 `export MIVO_PG_PASSWORD`(取 POSTGRES_PASSWORD 的值)。
2. **`pm2 restart <name>` 不重载 ecosystem env**:改名/改 `MIVO_PG_MAX_CONNECTIONS`/`MIVO_ASSET_STORE_DIR` 后,`pm2 restart mivo-canvas` 沿用**旧** ecosystem env(新值不生效,9:00/17:00 自动部署有中断风险)。必须 `pm2 startOrRestart ecosystem.config.cjs --update-env` 才重载 env 块 + ecosystem。
3. **缺密码必须 fail visibly**:`MIVO_PG_PASSWORD` 缺 → `resolvePersistBackendConfig` 抛错 → BFF 启动停(不静默降级 memory 假绿)。

**密码注入约定(不入库)**:ecosystem.config.cjs 的 `env` 块只放非密 env;`MIVO_PG_PASSWORD` 不写在本文件(入 git)。deploy.sh(`CLAUDE.md` 部署规则,只在服务器不入仓库)负责:source `ops/postgres/.env` → `export MIVO_PG_PASSWORD`(取 `POSTGRES_PASSWORD` 的值,不打印/不落日志)→ `pm2 startOrRestart ecosystem.config.cjs --update-env`。

```bash
# ─── 可执行部署步骤(deploy.sh 内容;以 yanjian 身份)──────────────────────────
# 0. 前置:代码已 pull + build(git checkout main && git pull origin main && npm ci && npm run build)
set -euo pipefail
cd /AIGC_Group/mivo-canvas
# 1. 读 PG 密码:source ops/postgres/.env(载入 POSTGRES_PASSWORD;不打印/不落日志)
if [ ! -f ops/postgres/.env ]; then
  echo "FAIL: ops/postgres/.env missing — 须先配置 PG 密码(见 t1.1 runbook)" >&2
  exit 1
fi
# shellcheck disable=SC1091  # .env 仅服务器侧,不入仓库
set -a; . ops/postgres/.env; set +a
# 缺密码 fail visibly(不静默降级 memory):
: "${POSTGRES_PASSWORD:?FAIL: POSTGRES_PASSWORD 未设置 — BFF 启动会 fail visibly}"
# 2. 映射 POSTGRES_PASSWORD → MIVO_PG_PASSWORD(BFF 读后者,见 pgConfig.ts;值相同,名不同)
MIVO_PG_PASSWORD=${POSTGRES_PASSWORD}
export MIVO_PG_PASSWORD
# 3. 启动/重启 ecosystem(--update-env 重载 env 块 + 新 ecosystem;首启=start,已存=startOrRestart)
sudo -u yanjian env "MIVO_PG_PASSWORD=${MIVO_PG_PASSWORD}" \
  pm2 startOrRestart ecosystem.config.cjs --update-env
# 4. 存活 + /readyz 探活(等 200 再放流量;R2-4:全失败必 exit 1,非 rc=0 自然结束)
sudo -u yanjian pm2 list
ready=0
for i in $(seq 1 30); do
  if curl -fs http://127.0.0.1:8080/readyz; then ready=1; break; fi
  sleep 1
done
if [ "$ready" != 1 ]; then
  echo "FAIL: /readyz not ready after 30s — 不放流量,人工排查(见 runbook §2)" >&2
  exit 1
fi
```

**三场景验证(F6 验收)**:
| 场景 | 步骤 | 期望 |
|---|---|---|
| 空 PM2 首启 | deploy.sh 注入密码 + `pm2 start ecosystem.config.cjs --update-env` | mivo-canvas online;/readyz 200 |
| 密码轮换 | 改 `ops/postgres/.env` 的 `POSTGRES_PASSWORD` → 重跑 deploy.sh | BFF 重启后连新密码;旧连接断 |
| 参数变更(改 `MIVO_PG_MAX_CONNECTIONS` / `MIVO_ASSET_STORE_DIR` in ecosystem) | 重跑 deploy.sh(`--update-env`) | `pm2 jlist` 见新值;`pm2 restart mivo-canvas`(无 --update-env)**不**生效,禁用 |
| 缺密码 | 删 `ops/postgres/.env` 或 `POSTGRES_PASSWORD` 空 → 重跑 deploy.sh | step 1/2 fail visibly(exit 1),不静默降级 memory |

> **⚠ 待 lead 拍板(本 worker 不动 CLAUDE.md/AGENTS.md)**:CLAUDE.md「部署规则」与 AGENTS.md 的四步(`pull→ci→build→pm2 restart mivo-canvas`)需同步为「`pm2 startOrRestart ecosystem.config.cjs --update-env` + `POSTGRES_PASSWORD→MIVO_PG_PASSWORD` 映射」。该同步由 lead 决策执行(本 runbook 仅标注,不改这两个文件)。

## 2. /readyz 端点(区别于 /healthz)

> 产物:`server/app.ts` 挂 `/readyz` + `server/lib/readiness.ts` + 路由测试 `server/__tests__/readyz.test.ts`。

| 端点 | 语义 | 返回 | 谁用 |
|---|---|---|---|
| `GET /healthz` | liveness — "进程活" | 恒 200 `{status:'ok'}` | pm2/docker HEALTHCHECK(进程死即摘) |
| `GET /readyz` | readiness — "依赖此刻可用" | 全 ok → 200;任一 fail → **503** + 各 check 诊断体 | 部署/网关摘流量;pm2 重启后探活 |

`/readyz` 检查项(响应体 `{status, persist, assetDir}`):
- **persist**:调 `backend.ping()`。memory 恒 ok;PG 跑 `SELECT 1`(连接池探活,受 `connectionTimeoutMillis` 排队超时保护)。
- **assetDir**:`MIVO_ENABLE_ASSET_SERVICE=1` 时探写一个 `.readyz-probe-*` 文件再删;service 关时 `skipped`(不参与 fail 判定——不依赖该卷就不该因它摘流量)。

```bash
# 部署后探活(服务器本地):
curl -s http://127.0.0.1:8080/readyz | jq .        # 预期:status=ok → HTTP 200
curl -s -o /dev/null -w '%{http_code}\n' http://127.0.0.1:8080/readyz   # 预期:200
# PG 挂时:HTTP 503,body.status=degraded,body.persist.status=fail + reason
# pm2 重启后探活(等 readyz 200 再放流量;R2-4:全失败必 exit 1):
ready=0
for i in $(seq 1 30); do
  if curl -fs http://127.0.0.1:8080/readyz; then ready=1; break; fi
  sleep 1
done
if [ "$ready" != 1 ]; then echo "FAIL: /readyz not ready after 30s" >&2; exit 1; fi
```

> 关键不变量:`/healthz` 不随 `/readyz` degraded 变(进程活≠依赖可用);网关/部署应探 `/readyz` 决定是否放流量,探 `/healthz` 决定是否重启进程。

## 3. 连接预算(PG pool)

> 产物:`server/persist/pgConfig.ts`(`connectionTimeoutMs` 字段 + env)+ `server/persist/pgBackend.ts`(Pool 传 `connectionTimeoutMillis`)+ ecosystem.config.cjs env 固化。

### 3.1 三参数(pg Pool)

| 参数 | env | 默认 | 作用 |
|---|---|---|---|
| `max` | `MIVO_PG_MAX_CONNECTIONS` | 10 | 池上限:同时打开的 PG 连接数。单实例保守 10 |
| `idleTimeoutMillis` | `MIVO_PG_IDLE_TIMEOUT_MS` | 30000 | 空闲连接 30s 后关(释放 PG 连接) |
| `connectionTimeoutMillis` | `MIVO_PG_CONNECTION_TIMEOUT_MS` | 5000 | 池满时排队等待 5s,超时即抛(fail fast,不无限排队拖垮 BFF) |

### 3.2 容量预算(单实例 → 多实例演进)

- **单实例 BFF(当前,灰度)**:`max=10`。PG compose 限 `max_connections` 默认 100(postgres:16-alpine 默认);BFF 占 10,留 90 给备份/运维查询/未来多实例。安全。
- **多实例协作(T1.4+)**:`Σ(各实例 max) ≤ PG max_connections × 70%`。例:4 实例 → 每实例 max=15(Σ=60 ≤ 70)。调 PG compose 的 `max_connections`(runbook §资源限制)配合。
- **BFF 单实例假设**:PG backend 用 in-process 全局索引缓存(`projectIndex`/`canvasIndex` 启动预热);**改 cluster 多实例前必须先解决缓存一致性**(见 `docs/decisions/pg-backend-schema.md §4`)。ecosystem `exec_mode=fork` + `instances=1` 是该假设的强制表达。
- **fail fast 取舍**:`connectionTimeoutMillis=5000` 让池满时快速报错(→ /readyz 503 → 摘流量),而非无限排队让 BFF 表面活着实则请求全卡。监控 `persist.status=fail` 即需扩容 max 或排查慢查询。

## 4. 固定 asset dir(杜绝随 cwd 漂移)

> 产物:`server/lib/assetStore.ts`(`resolveAssetStoreDir` env 驱动)+ ecosystem.config.cjs(`MIVO_ASSET_STORE_DIR` 固定)。

**问题**:BFF 用 `cwd` 解析 `dist/` 等;若 asset 存储也随 cwd,部署脚本从不同目录起 pm2 时 blob 路径漂移 → 旧图 404。
**解法**:`resolveAssetStoreDir()` 读 `MIVO_ASSET_STORE_DIR`,默认 `~/.mivo-canvas/assets`(仓库外,不漂移到 git 工作树)。**生产用 ecosystem.config.cjs 显式固定**:

```
MIVO_ASSET_STORE_DIR=/AIGC_Group/mivo-canvas-data/assets
```

- 路径在持久卷(`/AIGC_Group/mivo-canvas-data/`,与 PG 数据/备份同卷族),不随仓库 cwd 漂移。
- `/readyz` 探写该目录确认卷挂载正常 + 权限对(见 §2)。
- 部署前一次性创建:`sudo -u yanjian mkdir -p /AIGC_Group/mivo-canvas-data/assets`(yanjian 持有)。

## 5. restore 含业务行 + asset(扩展 t1.1 restore drill)

> 产物:`ops/postgres/restore-drill.sh`(加 `--dry-run` + 业务行抽查 + asset 一致性)+ `ops/postgres/backup.sh`(加 asset dir 快照,可选)。
> 与 server/routes/projects.ts 的 restore saga 补偿无关(另一 worker 负责,本文件不动 BFF restore 逻辑)。

### 5.1 backup.sh 扩展(asset dir 快照 + F3 manifest)

backup.sh 新增(P0.3 返修):
- **`ASSET_BACKUP_DIR`**(默认与 PG 同卷 `.../asset-backups`):开启后 `tar` 一份 asset dir 到 `asset-snap-<ts>.tar.gz`。**F4 返修:开启时 snapshot 成功 = hard gate**——`ASSET_STORE_DIR` 缺失或 tar 失败 → backup `exit 1`(不 ship 一个 asset 侧缺失的 dump;旧实现仅 WARN 非阻断)。默认不开启(向后兼容)。
- **R2-3 备份绑定**(服务开⇒备份必配):生产 ecosystem 固定 `MIVO_ENABLE_ASSET_SERVICE=1`,backup.sh 据此把"服务开⇒备份必配"绑成 hard gate——`MIVO_ENABLE_ASSET_SERVICE=1` 且 `ASSET_BACKUP_DIR` 空 → `exit 1`(fail visibly,fail-fast 在 docker 之前)。防 cron 裸跑产出"只含 PG dump 不含 asset 快照"的假闭环备份(灾备恢复时 asset dir 缺失)。cron 段须同时导出这两个 env(见 t1.1 runbook §2/§6)。
- **F3 manifest sidecar**(`<dump-basename>.manifest.json`):同批写三核心表(persist_records/projects/canvases)行数 + `assetSnap` 名 + `assetBlobCount`;restore-drill 据此同批配对 + 行数比对。与 `.dump` 同保留策略滚动清理。**R2-2 返修:行数从 dump 快照导出(`pg_restore --data-only --table` 计 COPY 行),非 live DB 查询**——杜绝 dump/计数分时取的并发写 race 把好恢复判 mismatch。

```bash
# 开启 asset 快照(服务器 cron;MIVO_ENABLE_ASSET_SERVICE=1 配合 R2-3 绑定):
MIVO_ENABLE_ASSET_SERVICE=1 ASSET_BACKUP_DIR=/AIGC_Group/mivo-canvas-data/asset-backups ./ops/postgres/backup.sh
# 若只设 MIVO_ENABLE_ASSET_SERVICE=1 不设 ASSET_BACKUP_DIR → FAIL exit 1(R2-3 绑定:防假闭环)
```

### 5.2 restore-drill.sh 扩展(F3/F4/F8 + R2-2 硬闸门)

| 项 | 行为 | 成功标准(P0.3 + R2-2 返修) |
|---|---|---|
| `--plan-only` | 只打印计划,不碰 docker/PG/dump | exit 0(F8:无 dump 亦成功,纯脚本逻辑校验用) |
| `--dry-run` | 有 dump **真跑** `pg_restore -l`(TOC 预校验);无 dump → FAIL(R2-6:空结果正常化走显式分支) | TOC 可读 → exit 0;损坏/无 dump → exit 1(F8:不许假绿;旧实现无条件 exit 0) |
| F3 核心表 hard gate | 恢复后三核心表(persist_records/projects/canvases)须存在 | 缺一 → exit 1(R2-2 矩阵:`ALLOW_EMPTY_SCHEMA=1` 只许**三表全缺 + manifest 明示 pre-schema**;部分缺 / 全缺未声明 → 仍 exit 1;生产禁用此开关) |
| F3 manifest 行数比对 | 恢复后逐表行数 vs manifest(**R2-2:从 dump 快照导出的 count,非 live DB 查** → 无并发写 race) | 不等 → exit 1(restore 损失数据);旧备份无 manifest → WARN 跳过;**manifest 存在但 no-jq / 坏 JSON → exit 1**(R2-2 hard fail,不再当 -1 跳过假绿) |
| F4 asset drill | 真解压 snapshot + sha256 vs 文件名(内容寻址)+ **R2-3 .meta.json 内容校验**(JSON 可解析 + contentHash=实 hash + sizeBytes=实大小 + 必要字段)+ manifest blob 数比对 | 损坏 tar/哈希不匹配/缺 metadata/**metadata 内容坏**/blob 数不等 → exit 1(开启时为 hard gate;关 → skipped) |

**R2-2 矩阵自动验收**:`bash ops/postgres/test-restore-drill-matrix.sh`(无 docker/PG,26 场景:allow0/1 × 全缺/部分缺/全在 × manifest valid/malformed/no-jq + count_copy_rows 纯函数)。

**本地实跑(无需 docker/PG)**:
```bash
cd <worktree>
./ops/postgres/restore-drill.sh --plan-only                       # 预期:打印 plan + exit 0(无 dump 亦成功)
./ops/postgres/restore-drill.sh --dry-run                         # 无 dump → exit 1(F8:不许假绿)
# F4 asset 校验单测(造合成 tar:valid/corrupt/hash-mismatch/missing-meta/count-mismatch):
bash ops/postgres/test-asset-verify.sh                            # 预期:PASS=7 FAIL=0
```

> F8 返修:`--dry-run` 不再无条件 exit 0——有 dump 必真跑 `pg_restore -l`(损坏 → exit 1);`--plan-only` 是唯一允许无 dump 成功的模式。本地无 docker 时 `pg_restore -l` 走本地 `pg_restore` 二进制(若有),否则 `--dry-run` 因无法校验 TOC 而 exit 1(非假绿)。

**生产 restore drill(含业务行 + asset)**:见 t1.1 runbook §3;restore-drill.sh 现额外跑 F3 核心表 hard gate + manifest 行数比对 + F4 asset 真解压校验,结果进 restore-drill.log。

## 5.4 剩余风险(P0.3 返修如实标注,不实现)

双审认定以下两项为已知残余风险,本 runbook 如实标注,不在本 PR 实现(留生产上线后据实决策):

1. **`max_memory_restart=1G` 无 RSS 基线**:1G 为保守估值,未实测生产峰值 RSS。旧表述"BFF+Leafer"不实——Leafer 跑在浏览器(客户端),不在 pm2 BFF 进程内,不计入此 RSS;BFF 实际内存占用 = Node + Kysely + pg + Hono + in-process 索引缓存。生产上线后采一轮峰值(连真实数据量 + 多用户并发),据实调 `max_memory_restart`。
2. **asset snapshot 与 dump 分时缺一致时间点**:pg_dump(逻辑快照,事务一致)与 asset dir tar(文件级,非冻结)分时取,二者无统一时间点;灾备恢复时可能 asset 比 dump 略新或略旧。缓解:backup.sh 写同批 manifest(`assetSnap` + `assetBlobCount` 配对 dump),restore-drill 据此同批校验。彻底解决需 asset 写短冻结或 manifest 边界声明,留后续。
