## 1. 切换日总览(五步顺序 + 依赖)

```text
切换日(单个 user 维度,逐 user 跑):
  ① 导出该 user 的本地 IDB → migration bundle JSON        (前置:0.1/0.4/0.7)
  ② 冻结写入窗口(服务端 maintenance + 客户端 read-only)   (前置:0.2)
  ③ ingest JSON → 服务端 PG(T1.3 API,按 record 分片避 1MB)  (前置:0.2/0.3)
  ④ 校验:数量 + 抽样字段比对(IDB vs PG)                    (前置:③ 完成)
  ⑤ 翻默认 kernel=new + stale-client 门控                  (前置:④ pass)
  → 该 user 切换完成;逐 user 重复。全量完成后翻默认(§6)。
```

> 单 user 切换 = 一次 ①→⑤。多 user 可串行(安全)或按 owner 批准并行(lead 拍)。**任何一步失败 → 走 §5 回滚,不许跳步往前推**。

### 2.2 选项

| 选项 | 入口 | 优势 | 劣势 |
|---|---|---|---|
| **(a) 临时 UI 入口** | Settings 加 flag-gated "切换日导出" 按钮,或 `?export=migration` 路由,调 `exportUserMigrationBundle()` → 下载 JSON | 用户自触发、可审计、复用 UI 基建;operator 不必碰 DevTools | 临时 UI 要走 PR + flag + 切换后清理 |

| **(c) 复用 v10 迁移仪式** | 借 `persistMigration.ts` 的 dry-run + 快照 + rollback + `MigrationResult` 模式套到 IDB→server 路径 | 切换可逆、可审计、与 §4 回滚同源 | 仪式可借鉴,**transport 不可复用**(persistMigration 是 IDB→IDB 结构迁移,非 IDB→server 跨系统) |

### 2.3 推荐 + 理由(已裁决 2026-07-10,按推荐)

**推荐:(a) 临时 UI 入口 + 借鉴 (c) 仪式**。理由:
1. IDB 在用户浏览器,服务端够不着 — **必须用户自触发导出**;(a) 比 (b) 干净(不必让 operator 碰 DevTools,可审计、有 flag、切换后删)。
2. 批量原语已就位(`rawIdbStorage` PR #188 建、#189 改名),`exportUserMigrationBundle()` 几乎零新增:读 canvas blob + chat blob + 遍历 asset store → 一个 JSON。
3. (c) 的 dry-run/snapshot/rollback/MigrationResult 套到 per-user 切换 = §4 回滚同源,一套仪式贯穿。
4. (b) 作 fallback:若 lead 判临时 UI 太重,降级为 dev-only `/api/debug/idb-dump` 路由(flag + 鉴权 + 切换后删),仍调同一 `exportUserMigrationBundle()`。

> **已裁决(2026-07-10,按推荐,D-1/D-2)**:① 走 (a)+(c) —— 临时 UI 入口 + 借鉴 v10 仪式;临时 UI 入实施 PR 另开(本 runbook 是文档);ingest 脚本落 `scripts/migration/`。

## 5. ④ 回滚预案(借鉴 persistMigration 仪式)

**前置**:任何一步(②/③/④)失败即触发;IDB 源数据未删(④ 的回滚源)。

**回滚策略**(直接套 `persistMigration.ts` 仪式):
```text
persistMigration 仪式(src/kernel/persistMigration.ts):
  dryRunMigration   (零写,诊断)        → 本 runbook §2.4 的 --dry-run
  migrateV10ToV11    (快照 ckpt → 写)   → §2.4 ingest 前先 snapshot 服务端该 user 现状
  rollbackFromV11    (从快照恢复)        → 本 §5 回滚
  MigrationResult{ok,error}              → 每步返回结构化结果,不静默
```

**操作**:
```bash
# 5a 服务端回滚:该 user 的迁移写入回滚(软删或物理删迁移进来的 record)
#   方式 A(推荐,软删):标 is_deleted=true(对齐 soft-delete-semantics §2,可 restore)
#   方式 B(物理删,谨慎):DELETE 该 user 在切换窗口内新写入的 record
```

## 6. ⑤ stale-client 处理(旧版本页面还开着的用户)

### 6.1 现状(FX-4 + 版本检测,源码事实)

| 现状 | file:line | 含义 |
|---|---|---|
| 1MB 硬上限 + 清洁 413 | `server/lib/config.ts:35` `jsonRequestMaxBytes=1048576`;`server/lib/request.ts:43` `readBodyWithLimit`;多路由 413 处理;contract test `server/__tests__/p1c.test.ts:550` | 整 doc PUT 超 1MB 返 413;FX-4 解法 = 节点级 PATCH(与 revision 同设计),非整 doc |
| 版本注入点(骨架) | `src/store/remoteDebugReporter.ts:85` `appVersion = import.meta.env.VITE_MIVO_VERSION \|\| '0.0.0'` | 版本注入骨架在,但值是 `0.0.0` |
| package.json 版本 | `"version": "0.0.0"` | 非真实版本,无 CI 注入 |
| 版本门控/stale 检测 | **无** | 全仓无 client version check / API version gate / stale-client 提示 |

> 切换日后 stale-client 风险:用户旧 tab 还开着(kernel=legacy,数据写本地 IDB),但服务端已是真相 → 写丢到本地、读从服务端,split-brain;旧 tab 整 doc PUT 大画布 → 413 刷屏。

### 6.2 选项

| 选项 | 机制 | 优势 | 劣势 |
|---|---|---|---|
| **(a) 强刷提示** | CI 注入真实版本(`VITE_MIVO_VERSION=<git-sha/tag>` → `vite.config.ts`);客户端比对 build 版本与服务端最新,旧版弹非-dismissible "版本已更新,请刷新" | UX 层友好;防用户继续操作旧 UI | 只 catch 会刷新的 tab;切窗口间打开的 tab 抓不住;不拦写入 |
| **(b) API version gate** | 服务端写路由要求 `X-Mivo-Build` header(或 min-version 校验),低于 min 返 426 Upgrade Required + 刷新指令 body | **服务端硬拦** stale 写,防 split-brain + 防 413 刷屏;catch 所有 tab(含不刷新的) | 需 T1.3 写路由落地;旧 client 不带 header 的兼容窗口(灰度) |
| **(c) 切换窗口冻结即兜底** | §3 冻结期服务端 503 拒所有写,旧 tab 写全失败 | 零新增(复用 §3) | 只覆盖冻结窗口;窗口后旧 tab 仍 stale |

### 6.3 推荐 + 理由(已裁决 2026-07-10,按推荐)

**推荐:(b) API version gate + (a) 强刷提示作 UX 层 + (c) 冻结期兜底**。理由:
1. 唯一能拦住"页面还开着不刷新"的 stale tab 的是**服务端硬门控**(b);(a) 只 catch 会刷新的 tab。
2. (a) 是 (b) 的 UX 兜底:gate 返 426 时客户端弹强刷提示,不只静默失败。
3. (c) 是切换窗口内的零成本兜底(复用 §3),窗口后靠 (b) 续航。
4. 前置小:CI 注入真实版本(`VITE_MIVO_VERSION` 从 git SHA,改 `vite.config.ts` + CI)是几十行;(b) 的 `X-Mivo-Build` 校验随 T1.3 写路由一并落。

> **已裁决(2026-07-10,按推荐,D-3/D-4)**:(b) 随 T1.3 写路由一并落;(c) 冻结期兜底复用 §3;min-version 灰度 = 旧 client 无 header 窗口期放行。

### 6.4 操作(前置 / 命令 / 预期 / 失败处理)

**前置**:0.2(T1.3 写路由)就绪;CI 注入真实版本(`VITE_MIVO_VERSION` 非 `0.0.0`)。

**翻默认 kernel + 挂 version gate**:
```bash
# 6a 翻默认 kernel:legacy → new。翻转点 = src/app/kernelMode.ts 的 isLegacyKernel 模块常量
#   (canvasPersistConfig.ts:29 据它选 idbStateStorage(legacy) vs docKernelPersistStorage(new));
#   翻默认 = 改 kernelMode 的 ?kernel= 解析默认值(或环境旗标覆盖,机制待 T1.3 确认)
#   注意:kernel=new(#189)只切本地三域 canonical,≠ 服务端真相 —— 服务端真相靠 T1.3 syncToServer 接线 + §2 搬迁
sudo -u yanjian MIVO_KERNEL_DEFAULT=new pm2 startOrRestart /AIGC_Group/mivo-canvas/ecosystem.config.cjs --update-env   # --update-env 重载旗标(旗标机制待 T1.3 确认;裸 restart 不重载 env)
# 6b 挂 version gate:写路由校验 X-Mivo-Build >= min(灰度:不带 header 的旧 client 先放行窗口期)
#   TODO-T1.3: gate 逻辑随 T1.3 写路由落;min-version 与灰度策略 lead 拍
```
预期:刷新用户进 kernel=new(服务端真相);旧 tab 写请求被 426 拦 + 弹强刷提示;冻结期(§3)写全 503。
失败处理:6a 翻默认后用户 hydrate 失败(服务端无该 user 数据)→ 该 user 未走 §2 搬迁,**回滚 6a** 把默认翻回 legacy,补跑该 user 的 ①→⑤;6b 误拦新 client(header 缺/版本比错)→ min-version 设错或 CI 注入失败,先关 gate 回灰度放行,修版本注入;旧 tab 不弹提示只静默 426 → (a) 强刷提示未接线,补 UX。
