## 0.1 宿主端口决策(共享机冲突)

> 共享机 10.102.80.15 上 5432 已被 `xd-aigc-artdam-postgres` 占用(另有 pgvector 占 55432/55433)。
> MivoCanvas PG 走 **55442**(空闲,与既有实例错开);容器内端口恒 5432,仅宿主映射改。

**机制**:`docker-compose.yml` ports 为 `127.0.0.1:${MIVO_PG_HOST_PORT:-5432}:5432` —— 宿主端口走 env,默认 5432(向后兼容),容器内恒 5432;healthcheck 走容器内 `pg_isready`,不受宿主端口影响。

**本部署 .env 取值**(owner 在服务器 `ops/postgres/.env` 填):
```
MIVO_PG_HOST_PORT=55442
```

**端口占用自查**(目标宿主端口代入):
```bash
ss -tlnp | grep 55442     # 有输出=被占;无输出=空闲
# 或:lsof -iTCP:55442 -sTCP:LISTEN
```

**BFF 连接 PG**:
- BFF 与 PG 同机 → 走 `127.0.0.1:55442`(宿主映射端口,非容器内 5432)。
- 连接配置项名按 `server/` 现有 PG 连接约定。**经查 `server/` 目前无 PG 连接串/env**(`server/lib/config.ts` 仅覆盖 image/LLM/platform/asset 配置,全 `server/` 无 postgres/PGHOST/PGPORT/DATABASE_URL 引用)。
- → **T1.3 PersistBackend PG 实现接线时**,BFF 连接串端口字段读 `MIVO_PG_HOST_PORT`(默认 5432,本部署 55442),host `127.0.0.1`,库名/用户/密码沿用 `POSTGRES_DB`/`POSTGRES_USER`/`POSTGRES_PASSWORD`。T1.1 不动 BFF 连接代码。

## 0.2 网段决策(防 2026-07-11 事故复发)

> 2026-07-11 `docker compose up` 重建 mivo-postgres 时,compose 自动建的 `postgres_default` 网络抢占 **172.26.0.0/16**,与公司边缘网关(172.26.164.81)网段冲突 → 已登录用户 502 近 11h(详见 §9)。
> **防复发**:`docker-compose.yml` 显式声明 `networks.default` + 固定 ipam 网段,网络名固定 `mivo-pg-net`,网段固定 `192.168.228.0/24`。compose down/up 后网络名/网段均不变,杜绝自动选段冲突。

**选段依据**(为什么是 `192.168.228.0/24`):
- 避开 `172.16.0.0/12`(172.16-31.x):公司在用 172.26.x(边缘网关 172.26.164.81),docker 默认用户桥池也从该 /12 分配(事故正是抢到 172.26.0.0/16);
- 避开 `10.0.0.0/8`:服务器网段 10.102.x(见 §0.1 共享机 10.102.80.15);
- 避开 `172.17.0.0/16`:docker0 默认桥;
- 选 `192.168.228.0/24`:`/24`(254 可用地址)足够单容器;228 段冷门,远离常见 `192.168.0/1` 办公/家庭段,进一步降冲突概率。

**上服务器前的硬性检查**(owner 实操前必跑,见 §0 表 0.8 行):
```bash
ip route | grep 192.168.228.0 || echo "ip route: 宿主路由无本网段 — OK"
# 遍历所有 docker 网络的 IPAM subnet(不只看 mivo-pg-net;同名/异名网络都得查,否则同网段异名漏检):
docker network inspect $(docker network ls -q) --format '{{.Name}}: {{range .IPAM.Config}}{{.Subnet}}{{end}}' 2>/dev/null | grep 192.168.228.0 \
  && echo "WARN: 既有 docker 网络已用本网段(含异名)" || echo "docker networks: 无本网段 — OK"
```
任一命中 → 先 `docker network rm <name>` 清残留(含旧 `postgres_default`,见 §9.4);仍冲突 → 选另一冷门 /24(同步改 `ops/postgres/docker-compose.yml` + §0 表 0.8 + §9)。

> ⚠️ **通用规则(非仅 PG)**:任何新 compose 服务上共享服务器前,必须先 `ip route` 对照公司网段(172.26/10.102 等)与 docker 既有网络,固定 ipam 网段后再 up。默认自动选段在共享机上等于埋雷。

## 3. restore drill 步骤(首次必跑留证,plan §4 验收)

```bash
cd /AIGC_Group/mivo-canvas
chmod +x ops/postgres/backup.sh ops/postgres/restore-drill.sh   # 首次赋可执行

# 3.1 先手跑一次 backup(确保有 .dump 可恢复)
./ops/postgres/backup.sh
ls /AIGC_Group/mivo-canvas-data/backups/*.dump   # 预期:有 mivocanvas-<ts>.dump(空库约 825 字节)
tail /AIGC_Group/mivo-canvas-data/backups/backup.log   # 预期:ok: pg_dump (N bytes) + === backup done ===
# dump 非空校验:backup.sh 校验 dump > 100 字节(空库 schema 头实测 825 字节);≤ 阈值 → `FAIL: pg_dump produced N bytes` + 非零退出(不再静默当成功)。

# 3.2 跑 restore drill(恢复到临时库 mivocanvas_drill,跑完即 DROP,不碰生产)
./ops/postgres/restore-drill.sh
tail /AIGC_Group/mivo-canvas-data/backups/restore-drill.log
# 预期最后一行:=== restore drill done: PASS ===

# 3.3 归档演练证据(plan §4 "restore drill 实跑一次留证")
cp /AIGC_Group/mivo-canvas-data/backups/restore-drill.log \
   /AIGC_Group/mivo-canvas/history/restore-drill-$(date +%Y%m%d).log
```
成功标准(restore-drill.sh 内已实现):dump TOC 可读预校验(`pg_restore -l`)+ `pg_restore` 退出 0 + `SELECT 1` 通 + public 表计数为数字 ≥ 0(空库合法,BFF schema 未迁移前为 0)。0 字节/损坏 dump 在预校验即 FAIL,不空跑报 PASS。

## 4. RTO/RPO 目标(建议值 + 依据)

| 指标 | 建议值 | 依据 / 限制 |
|---|---|---|
| **RPO**(可容忍数据丢失) | **≤ 24h** | 每日 03:00 pg_dump;最坏丢上次备份→故障点之间的变更(≤ 24h)。canvas 数据以节点级 revision 合并(plan §13.5),单次丢 24h 改动可由协作端重放补;但 lead/owner 若判不可接受,见 §7 加 WAL 归档(PITR)→ RPO ≤ 5min。 |
| **RTO**(恢复到可用) | **≤ 30min**(pg_restore 主路径) | pg_restore 从 .dump 恢复;canvas PG 规模 < 1GB 时实测分钟级。若 .dump 损坏,走数据目录 tar.gz 快照路径:PG 停 → 解压 → 启动 → crash-recovery,RTO ≤ 2h(快照是 crash-consistent,可能需 WAL replay)。**注**:快照需 root 或 postgres 身份产出(见 §8);当前服务器以 yanjian 跑快照会 WARN 失败,该 fallback 暂不可用,需先具备快照产出条件。 |

> 这两个值是建议,lead/owner 按业务 SLA 调;调了回填本表。

## 7. 扩展(非 T1.1 范围,记录给后续)

- **WAL 归档(PITR)**:RPO 从 24h 降到 ≤ 5min。在 compose 加 `archive_command`(wal归档到 backups/wal/)+ `restore_command`;`pg_basebackup` 做全量基线。归 T1.1 后的运维增强,不阻塞 T1.2a schema。
- **业务表断言**:BFF schema 落地(T1.3 后)后,restore-drill.sh §3c 加 `SELECT count(*) FROM <core_table>` 断言,确保恢复的不只是结构还有数据。
- **监控/告警**:cron 失败 → backup.log 末行非 `=== backup done ===` + 非零退出;接入 maker 告警或单独监控(见 CLAUDE.md maker 定时任务机制)。

## 8. 取舍说明

- **目录选址**:新建顶层 `ops/`(仓库此前无 ops/ 惯例;既有 runbook 在 `scripts/changelog/RUNBOOK.md` 是 changelog 专用)。ops/postgres/ 放 PG 运维产物( compose + 脚本),docs/runbook/ 放跨任务 runbook——按 lead 指定走,与既有 changelog RUNBOOK 分离,避免混进 scripts/。
- **认证**:compose 不设 `POSTGRES_HOST_AUTH_METHOD`,用镜像默认(local socket trust,docker exec 脚本免密;TCP scram,BFF 带密码)。仅绑 127.0.0.1 + 服务器受控访问,local trust 可接受;若需更严,设 `POSTGRES_HOST_AUTH_METHOD=scram-sha-256` + 给 backup/restore 脚本传 `PGPASSWORD`(脚本已用 docker exec 走 local,改 scram 后需补)。
- **资源限制**:用 `deploy.resources.limits`(Compose v2 的 `docker compose up` 落实);cpus=2/memory=2g 是 canvas app PG 的保守值,owner 按服务器规格调。
- **数据目录快照**:文件级 tar.gz,非块级/非 PITR;CHECKPOINT 降 crash-consistent 风险但不消除;pg_dump 是主备份(逻辑一致),快照仅灾备全量兜底。**诚实标注(2026-07-11 服务器实测)**:PG 数据目录由容器内 postgres 用户持有(mode 0700),宿主侧 `tar` 以 yanjian 身份读不了 → 当前服务器以 yanjian 跑快照会 WARN 失败、不产出快照文件(非阻断,pg_dump 主备份不受影响)。快照为**可选/需 root**:要产出可用快照,须 `sudo tar` 或改 `docker exec` 内 tar 走 postgres 身份(留后续增强,见 §7)。backup.sh 如实记 WARN,不假成功。
- **dump 非空校验(2026-07-11 修)**:backup.sh 在 pg_dump 后校验 `DUMP_SIZE > DUMP_MIN_BYTES`(默认 100,env 可覆盖)。pg_dump -Fc 空库 schema 头实测 825 字节,真损坏/0 字节才 ≤ 阈值 → `FAIL` + 删坏文件 + 非零退出。修此前的静默失败 bug:0 字节 dump 被当成功(只报 size 不校验)。restore-drill.sh 同步加 `pg_restore -l` TOC 预校验,0 字节/损坏 dump 在 drill 即 FAIL,不空跑报 PASS。
- **本地未跑 docker compose config**:本机无 docker(沙箱),只做 YAML 语法校验(python yaml 通过)+ shellcheck 两个脚本通过(backup.sh / restore-drill.sh 均 exit 0)。owner 在服务器跑 §6 第 4 步 `docker compose config` 做完整语义校验。

## 9. 事故复盘(2026-07-11 PG 重建网段抢占 → 502,11h)

> 2026-07-11 生产事故根因复盘与防复发措施。历史记录 + 操作规约,供后续运维参考,防同类故障复发。选段依据见 §0.2,固定 ipam 配置见 `ops/postgres/docker-compose.yml`。

### 9.1 时间线

| 时间(2026-07-11/12,UTC+8) | 事件 |
|---|---|
| 07-11 14:50 | `docker compose up` 重建 mivo-postgres。compose 自动创建 `postgres_default` 网络,docker 从用户桥池随机选段,分配到 **172.26.0.0/16**。 |
| 07-11 14:50 起 | `postgres_default` 网桥抢占 172.26.0.0/16 → 宿主到公司边缘网关(172.26.164.81)的路由被 docker 网桥黑洞。服务器无法回连 auth/网关做会话校验。 |
| 07-11 14:50 起 | mivo-canvas.dsworks.cn 对**已登录用户**持续 502(网关侧会话校验回连失败);**未登录用户** 302 正常(不需回连)→ 健康探测长期误判"应用正常"。 |
| 07-12 01:55 | lead 在服务器 `docker compose down` 释放 172.26.0.0/16 网段止血。PG 暂停(生产 BFF 自动降级用 memory 后端,零影响)。已登录用户 502 即时恢复。 |
| 07-12 ~02:00 | 根因定位:路由表对照 + docker 网络检查 → 确认网段抢占,非应用/SSO 侧故障。 |

**故障持续**:对已登录用户约 11 小时(07-11 14:50 → 07-12 01:55)。

### 9.2 根因

1. **compose 默认网络自动选段**:`docker-compose.yml` 未声明 `networks:` 时,compose 自动建 `<project>_default` 网络(本项目 project 名 = `postgres`,故网络名 `postgres_default`),docker 从 `172.16.0.0/12` 池随机分配 /16 子网。自动选段不感知宿主已有的企业内网路由,可能选到与之冲突的网段。
2. **抢占 172.26.0.0/16 与公司边缘网关(172.26.164.81)冲突**:docker 网桥在宿主路由表注入 172.26.0.0/16 → 宿主发往 172.26.x.x 的回连流量(会话校验)被导进 docker 网桥黑洞,网关侧收不到回连 → 已登录请求 502。
3. **故障极难从应用侧发现**:应用本地进程全绿(BFF 用 memory 后端降级零影响、未登录用户 302 正常),健康探测走未登录路径长期误判"正常"。表现是"回连方向断,应用本地全绿"——典型的网络层路由黑洞,非应用层故障,应用日志无异常。根因排查须从宿主路由表 + docker 网络入手,而非应用日志。

### 9.3 防复发

1. **固定 ipam 网段(本 PR 落地)**:`docker-compose.yml` 显式声明 `networks.default` + 固定 ipam(`192.168.228.0/24`,network name 固定 `mivo-pg-net`)。选段依据见 §0.2。compose down/up 后网络名/网段均不变,杜绝自动选段冲突。
2. **上服务器前 `ip route` 对照(硬性前置检查)**:任何新 compose 服务上共享服务器前,必须先 `ip route` + `docker network ls` 对照公司网段(172.26/10.102)与既有 docker 网络,确认无冲突后再 up。本检查已加入 §0 前置检查表 0.8 行。
3. **通用规则**:默认自动选段在共享机等于埋雷——所有共享机上的 compose 服务都应固定 ipam 网段,不依赖 docker 自动分配。

### 9.4 重启 PG 的正确命令(事故止血后的恢复路径)

> 止血时 lead 已 `docker compose down`(PG 暂停、网段释放)。恢复 PG 须在**本 PR 合并、服务器 pull 最新 main 后**执行(严禁直连服务器改代码,见 CLAUDE.md 部署规则):

```bash
# 以 yanjian 身份,在服务器仓库目录:
cd /AIGC_Group/mivo-canvas
git pull origin main                       # 拉取含本 PR 的最新 main(固定 ipam 配置已入库)

# 起前先跑 §0 表 0.8 行的网段冲突自查(防复发,遍历所有 docker 网络 IPAM,防同网段异名漏检):
ip route | grep 192.168.228.0 || echo "ip route: 宿主路由无本网段 — OK"
docker network inspect $(docker network ls -q) --format '{{.Name}}: {{range .IPAM.Config}}{{.Subnet}}{{end}}' 2>/dev/null | grep 192.168.228.0 || echo "docker networks: 无本网段 — OK"

# 清理止血时可能残留的旧网络:
docker compose --env-file ops/postgres/.env -f ops/postgres/docker-compose.yml down   # 删 mivo-pg-net(新 compose 声明的)+ 容器
# 旧 compose 自动建的 postgres_default 不在新 compose 声明里,down 不会删;若残留仍持 172.26.0.0/16 → 冲突未真正消除,须手动清(幂等,不存在即跳过):
docker network rm postgres_default 2>/dev/null && echo "已清理残留 postgres_default" || echo "postgres_default 未残留(止血 down 已清)— OK"

# 起 PG(现在用固定网络 mivo-pg-net / 192.168.228.0/24):
docker compose --env-file ops/postgres/.env -f ops/postgres/docker-compose.yml up -d
docker ps --filter name=mivo-postgres --format '{{.Names}} {{.Status}}'                # 预期:Up (healthy)
docker network inspect mivo-pg-net --format '{{range .IPAM.Config}}{{.Subnet}}{{end}}'  # 预期:192.168.228.0/24
docker exec mivo-postgres pg_isready -U mivo -d mivocanvas                              # 预期:accepting connections
```
确认 `mivo-pg-net` 子网为 `192.168.228.0/24` 且 PG healthy 即恢复完成;生产 BFF 切回 PG 后端由 T1.3 接线状态决定,不在本 runbook 范围。

### 9.5 备份 cron 在 PG 停用期间的表现(2026-07-11/12 止血窗口)

> PG 止血期间(07-11 14:50 down → lead 按 §9.4 恢复 up 前),每日 03:00 的 backup cron 会**fail-visibly**(预期行为,非静默失败):

- `backup.sh` 跑 `docker exec mivo-postgres pg_dump ...`,容器不存在 → `docker exec` 非零退出 → `backup.log` 记 `FAIL` + `cron.log` 记 stderr,**非零退出 + 可见日志**(`backup.sh` 的 dump 非空校验同样 fail-visibly,不假成功)。
- **无数据丢失**:PG 在事故前未正式承载生产数据(T1.3 schema 未迁移、未启用),且事故止血是 `down` 非 `rm`——数据卷 `/AIGC_Group/mivo-canvas-data/pg` 保留,容器重启即原样恢复。
- **PG 重新拉起后自愈**:按 §9.4 起回 PG 后,下一轮 03:00 cron 恢复成功,`backup.log` 末行回到 `=== backup done ===`。
- 止血窗口内 03:00 备份失败属**预期**,不触发告警升级;若 PG 已恢复后仍见 `FAIL`,则按 §3 restore drill 路径排查 dump/连接。
