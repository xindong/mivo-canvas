## 1. 安装步骤(每步:命令 + 预期输出)

### 1.1 创建数据/备份目录(以 yanjian 持有)
```bash
sudo mkdir -p /AIGC_Group/mivo-canvas-data/pg /AIGC_Group/mivo-canvas-data/backups
sudo chown -R yanjian:yanjian /AIGC_Group/mivo-canvas-data
```
预期:`ls -ld /AIGC_Group/mivo-canvas-data/pg` 显示 `yanjian yanjian`;**pg 目录必须为空**(PG 首次初始化要求,非空会拒跑)。

### 1.2 拉取仓库(获取 ops/postgres/ 产物)
```bash
cd /AIGC_Group/mivo-canvas
git pull origin main
ls ops/postgres/
# 预期:.env.example  backup.sh  docker-compose.yml  restore-drill.sh
```

### 1.3 生成 .env(填真实密码,不入库)
```bash
cd /AIGC_Group/mivo-canvas
cp ops/postgres/.env.example ops/postgres/.env
# 生成强密码:
openssl rand -base64 32
# 把输出粘到 ops/postgres/.env 的 POSTGRES_PASSWORD=
# 确认 .env 被 .gitignore 忽略(根 .gitignore: .env + .env.* 但 !.env.example):
git check-ignore ops/postgres/.env && echo ".env 正确忽略(不入库)" || echo "WARN: .env 未忽略!"
```
预期:`.env` 内容含真实 `POSTGRES_PASSWORD=...`;`git check-ignore` 输出 `ops/postgres/.env`(确认不入库)。

### 1.4 校验 compose 语法(不 up)
```bash
docker compose --env-file ops/postgres/.env -f ops/postgres/docker-compose.yml config
```
预期:打印解析后的 compose 配置(JSON-ish),无 error。若报 `POSTGRES_PASSWORD required` → .env 没填或路径错。

### 1.5 启动 PG
```bash
docker compose --env-file ops/postgres/.env -f ops/postgres/docker-compose.yml up -d
docker ps --filter name=mivo-postgres --format '{{.Names}} {{.Status}}'
```
预期:`mivo-postgres Up X seconds (healthy)`(`healthy` 来自 healthcheck,`start_period: 30s` 内转 healthy)。

### 1.6 验证 PG 连接
```bash
docker exec mivo-postgres pg_isready -U mivo -d mivocanvas
# 预期:/var/lib/postgresql/data/pg is accepting connections

docker exec mivo-postgres psql -U mivo -d mivocanvas -c "SELECT version();"
# 预期:PostgreSQL 16.x on x86_64 ...
```

## 2. cron 挂载(以 yanjian 身份,每日 03:00 错峰)

> CLAUDE.md:pm2 以 yanjian 为主;maker 定时任务 9:00/17:00。PG 备份错峰 03:00(无人值守低负载窗口)。
> yanjian 须在 docker 组(§0.4),cron 内 `docker exec` 才免 sudo。

```bash
# 以 yanjian 身份追加 cron(不破坏既有 crontab):
sudo -u yanjian crontab -l 2>/dev/null | { cat; echo "0 3 * * * /AIGC_Group/mivo-canvas/ops/postgres/backup.sh >> /AIGC_Group/mivo-canvas-data/backups/cron.log 2>&1"; } | sudo -u yanjian crontab -
# 验证:
sudo -u yanjian crontab -l | grep backup.sh
# 预期:0 3 * * * /AIGC_Group/mivo-canvas/ops/postgres/backup.sh >> .../cron.log 2>&1
```
说明:每日 03:00 北京时间跑 backup.sh;成功/失败日志进 `backup.log`(脚本内)+ `cron.log`(stderr)。

## 3. restore drill 步骤(首次必跑留证,plan §4 验收)

```bash
cd /AIGC_Group/mivo-canvas
chmod +x ops/postgres/backup.sh ops/postgres/restore-drill.sh   # 首次赋可执行

# 3.1 先手跑一次 backup(确保有 .dump 可恢复)
./ops/postgres/backup.sh
ls /AIGC_Group/mivo-canvas-data/backups/*.dump   # 预期:有 mivocanvas-<ts>.dump
tail /AIGC_Group/mivo-canvas-data/backups/backup.log   # 预期:=== backup done ===

# 3.2 跑 restore drill(恢复到临时库 mivocanvas_drill,跑完即 DROP,不碰生产)
./ops/postgres/restore-drill.sh
tail /AIGC_Group/mivo-canvas-data/backups/restore-drill.log
# 预期最后一行:=== restore drill done: PASS ===

# 3.3 归档演练证据(plan §4 "restore drill 实跑一次留证")
cp /AIGC_Group/mivo-canvas-data/backups/restore-drill.log \
   /AIGC_Group/mivo-canvas/history/restore-drill-$(date +%Y%m%d).log
```
成功标准(restore-drill.sh 内已实现):`pg_restore` 退出 0 + `SELECT 1` 通 + public 表计数为数字 ≥ 0(空库合法,BFF schema 未迁移前为 0)。

## 4. RTO/RPO 目标(建议值 + 依据)

| 指标 | 建议值 | 依据 / 限制 |
|---|---|---|
| **RPO**(可容忍数据丢失) | **≤ 24h** | 每日 03:00 pg_dump;最坏丢上次备份→故障点之间的变更(≤ 24h)。canvas 数据以节点级 revision 合并(plan §13.5),单次丢 24h 改动可由协作端重放补;但 lead/owner 若判不可接受,见 §7 加 WAL 归档(PITR)→ RPO ≤ 5min。 |
| **RTO**(恢复到可用) | **≤ 30min**(pg_restore 主路径) | pg_restore 从 .dump 恢复;canvas PG 规模 < 1GB 时实测分钟级。若 .dump 损坏,走数据目录 tar.gz 快照路径:PG 停 → 解压 → 启动 → crash-recovery,RTO ≤ 2h(快照是 crash-consistent,可能需 WAL replay)。 |

> 这两个值是建议,lead/owner 按业务 SLA 调;调了回填本表。

## 7. 扩展(非 T1.1 范围,记录给后续)

- **WAL 归档(PITR)**:RPO 从 24h 降到 ≤ 5min。在 compose 加 `archive_command`(wal归档到 backups/wal/)+ `restore_command`;`pg_basebackup` 做全量基线。归 T1.1 后的运维增强,不阻塞 T1.2a schema。
- **业务表断言**:BFF schema 落地(T1.3 后)后,restore-drill.sh §3c 加 `SELECT count(*) FROM <core_table>` 断言,确保恢复的不只是结构还有数据。
- **监控/告警**:cron 失败 → backup.log 末行非 `=== backup done ===` + 非零退出;接入 maker 告警或单独监控(见 CLAUDE.md maker 定时任务机制)。

## 8. 取舍说明

- **目录选址**:新建顶层 `ops/`(仓库此前无 ops/ 惯例;既有 runbook 在 `scripts/changelog/RUNBOOK.md` 是 changelog 专用)。ops/postgres/ 放 PG 运维产物( compose + 脚本),docs/runbook/ 放跨任务 runbook——按 lead 指定走,与既有 changelog RUNBOOK 分离,避免混进 scripts/。
- **认证**:compose 不设 `POSTGRES_HOST_AUTH_METHOD`,用镜像默认(local socket trust,docker exec 脚本免密;TCP scram,BFF 带密码)。仅绑 127.0.0.1 + 服务器受控访问,local trust 可接受;若需更严,设 `POSTGRES_HOST_AUTH_METHOD=scram-sha-256` + 给 backup/restore 脚本传 `PGPASSWORD`(脚本已用 docker exec 走 local,改 scram 后需补)。
- **资源限制**:用 `deploy.resources.limits`(Compose v2 的 `docker compose up` 落实);cpus=2/memory=2g 是 canvas app PG 的保守值,owner 按服务器规格调。
- **数据目录快照**:文件级 tar.gz,非块级/非 PITR;CHECKPOINT 降 crash-consistent 风险但不消除;pg_dump 是主备份(逻辑一致),快照仅灾备全量兜底。
- **本地未跑 docker compose config**:本机无 docker(沙箱),只做 YAML 语法校验(python yaml 通过)+ shellcheck 两个脚本通过(backup.sh / restore-drill.sh 均 exit 0)。owner 在服务器跑 §6 第 4 步 `docker compose config` 做完整语义校验。
