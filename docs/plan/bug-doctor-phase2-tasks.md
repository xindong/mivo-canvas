## P0 · 接单核心链路(顺序做,T2-1 是一切前置)

### T2-1 · Cindy Bot 接入与路由验证(二期预检的实弹化)
**做法**:①把 @Cindy 拉进 `#mivo-canvas`;②在 maker 里注册本机工作目录别名(hello 注册制,别名即白名单——对照 cindy 调研:server 只能派发已注册别名,本地路径不过网线);③频道里 @Cindy 发一条测试指令,验证任务经 Maker 转发落到**本机 mivo-canvas 工作目录的 maker 会话**;④验证回帖:会话 turn.end 后 Cindy 身份的答复出现在原消息串。全程记录到 unknowns-map 二期节(三验证逐项销号)。
**SC**:①@Cindy 测试单被本机会话接到(会话可见,workingDir=mivo-canvas);②Cindy 身份回帖落原消息串;③非注册目录的派发被拒(白名单生效证据);④断网/会话忙时的 ack 行为记录在案(accepted/queued/rejected 三态实测)。
**若验证不通**:停,回预检——这条通不过则二期接单整体重设计(回落自建监听方案),不硬上。

### T2-2 · "报 bug" 动词(接单→台账→回帖闭环)
**做法**:skill/prompt 工程(不建基建):@Cindy `报bug <描述>[截图]` → 接单会话识别意图 → 写台账"人工报告簇"(source=human-report,默认 S1 候诊,与日志簇同一条流水线、同分诊纪律)→ **3 秒内 ack 回帖**(队列位置+指纹号)→ 消息串 ts 作 externalKey 绑定簇(cindy 铁律:同 key 同会话,追问不开新单)→ 分诊/修复结果回帖原串。报单人白名单(初期:你+指定同事,存 notify.env)。**人工报告不自动高于日志信号**:同样过分诊核实,复现不出同样降 T3 建档。
**SC**:①白名单成员报 bug → ack 回帖含队列位置;②簇入台账且下一班车被分诊,结论回帖原串;③同串追问不新建簇(externalKey 幂等证据);④非白名单 @ → 礼貌拒绝不入队;⑤台账/看板同步可见该人工簇。

### T2-3 · "状态" 动词(看板 Slack 缩影)
**做法**:@Cindy `状态` → 回一张卡:队列 S0-S3 计数/进行中 PR/健康分/上轮班车时间——数据全部读现成 state.json/看板数据,零新数据源。
**SC**:①卡片字段与 localhost:8787 看板一致(同时刻对照);②任何白名单成员可查;③响应 <30 秒。

## P1 · 修复率与闭环体验(可与 P0 并行的独立线)

### T2-4 · stack trace 采集(修复率最大瓶颈,产品代码 PR)
**做法**:remoteDebugReporter 的 error 级上报附加 `error.stack`(截断 ≤2KB,剥 URL query/token);服务端 debug-records 脱敏管道同步扩展(stack 专用清洗:路径保留文件名+行号、剥用户目录前缀);契约"已知局限"第 2 条随之更新。**这是 src/store + server/lib 产品代码改动**——走完整评审流(sol 审 + 你审,重点脱敏正确性),不适用 T1 自动合并。
**SC**:①本地触发一个真实异常 → 服务端 JSONL 记录含脱敏后 stack(文件名:行号可见、无绝对路径无 token);②gate 指纹不受 stack 影响(指纹仍基于 message,stack 只进样本详情);③分诊 prompt 能拿到 stack 辅助定位(工作包样本含 stack 字段);④CI 全绿 + 双审通过合并;⑤上线一周后统计:含 stack 的簇复现成功率 vs 历史(进化轮回看)。

### T2-5 · 告警消息串闭环(S0 全生命周期同串可追)
**做法**:S0 告警发出时记录消息 ts 入台账(webhook 响应或 MCP message_ts);后续"修复中→PR→verified"全部回帖该串(cindy 范式:thread=事件锚点);依赖 bug-doctor webhook(身份独立后回帖才不混淆)。
**SC**:①一条 S0 从告警到 verified 的全部状态更新都在同一消息串;②频道主时间线只有告警首发一条(不刷屏);③台账里簇↔消息串 ts 双向可查。

### T2-6 · Slack 审批按钮("批准合并"进频道,安全敏感)
**做法**:T2 待审卡片加交互按钮(cindy 协议 v2 interaction.request,决策语义留 desktop:server 哑渲染,点击结果回传本机会话校验)——**安全门三条写死**:仅你的 Slack user id 可点有效;点击=等价"合"指令,仍走 merger 老流程(CI 绿+线程清零才真合);按钮只出现在 T2 卡,T0/禁区永不出按钮。
**SC**:①你点"批准"→ merger 被派→合并落地→原卡更新"已合并 by @你";②他人点击被拒并记录;③按钮卡在 desktop 侧有完整审计日志;④T0 告警卡确认无按钮。
**若 cindy interaction 通道不可用于第三方 loop**:降级为"回复关键词 `批 #PR号`"文本指令,SC 同等。

## P2 · 规模化(P0/P1 稳定后)

### T2-7 · 报单开放全团队 + 频控
**做法**:白名单扩到频道全员;每人每日报单上限(如 5)防刷;重复报单(同指纹)自动合并计数并回帖"已有工单 #指纹,+1";报单人订阅其簇的状态变更(修复/建档时 @报单人)。
**SC**:①非白名单成员(扩权后=频道成员)报单成功;②同 bug 两人报 → 一个簇两个 reporter,双方都收到结果回帖;③超频报单被温和限流。

### T2-8 · 播报礼仪细化
**做法**:告警 @你、战报不 @ 任何人、issue 公告带 GitHub 链接卡片、静默日一行合并进周摘要(降噪)。
**SC**:一周实际运行后频道消息抽查:无重复、无无效 @、静默日不刷屏。
